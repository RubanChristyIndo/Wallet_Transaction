# RESILIENCE.md

## Scenario: inventory moves to a separate service

Assume the item grant now goes through a separate inventory API reached over
HTTP, which can time out, fail outright, or process the same request twice —
and which cannot share a database transaction with the wallet/currency
store.

### The partial-failure window

Once currency and inventory are two systems, a single local ACID transaction
can no longer cover both. The dangerous window is exactly the gap between
"debit committed" and "grant confirmed": the process can crash, the network
call can time out, or the inventory service can accept the request but its
response never reach us — in all these cases we cannot know, from the
caller's perspective alone, whether the grant actually happened.

### Approach: transactional outbox + idempotent downstream call

1. **Local transaction (unchanged):** debit the wallet, insert the ledger
   row, **and insert a row into a new `purchase_outbox` table** — all in the
   same Postgres transaction as before. The outbox row includes a
   `purchase_id` (UUID, generated by us) and the intended payload (`itemId`,
   `playerId`). This transaction is atomic and durable exactly as described
   in DESIGN.md — the debit and "intent to grant" become durable together,
   or neither does.

2. **Async worker** polls `purchase_outbox` for rows in `status = pending`
   and calls the inventory service's grant endpoint, passing `purchase_id`
   as an idempotency key that the inventory service is required to honor
   (same pattern as our own `Idempotency-Key` header — the receiving service
   owns dedup on its side). On success, the worker marks the outbox row
   `status = granted` in a separate, short transaction.

3. **Retry safety:** if the worker crashes or the HTTP call times out before
   marking the row `granted`, the row is simply picked up again on the next
   poll. Because the call is idempotent on the inventory service's side
   (keyed on `purchase_id`), a duplicate call is safe — it either grants
   once or confirms it already did.

4. **Player-visible state:** the wallet is debited immediately (visible to
   the player right away), while the outbox worker eventually confirms the
   grant, typically within milliseconds to seconds. If the grant were to
   permanently fail (inventory service down for an extended period, item
   deprecated, etc.), a compensating action refunds the currency via a new
   ledger entry — never by mutating the original debit.

This is a **saga**, not a distributed transaction: two local commits (debit,
then eventually grant), linked by an idempotent async step, with a defined
compensation path if the second step can't complete.

### Why not two-phase commit (2PC)

2PC would require the inventory service to support an XA-style
prepare/commit protocol, which an external, independently-operated service
is unlikely to offer, and which introduces its own availability problems
(a coordinator crash can leave participants blocked holding locks). The
outbox pattern trades a small amount of eventual-consistency latency for
much better failure isolation and no cross-service locking.

## Sub-question: a bug double-granted currency to some players last week

### Detection, without downtime

The append-only `ledger` table (already part of this design, see
DESIGN.md) is exactly the audit trail needed here. `wallets.balance` is a
cached value; the ledger is the source of truth. A **reconciliation job**
run as a read-only background query (no locks that would affect live
traffic):

```sql
SELECT w.player_id, w.balance, COALESCE(SUM(l.delta), 0) AS ledger_sum
FROM wallets w
LEFT JOIN ledger l ON l.player_id = w.player_id
GROUP BY w.player_id, w.balance
HAVING w.balance <> COALESCE(SUM(l.delta), 0);
```

Any mismatch here indicates balance drift — including from a double-grant
bug, since a bug that doubled a credit would either (a) also double-insert
the ledger row, which shows up as an unusually large or duplicate-looking
ledger entry for that player, easy to `GROUP BY reason` and spot as an
anomaly, or (b) increment balance without a matching ledger row, which the
query above catches directly as drift.

This job runs continuously (e.g. hourly, on a read replica if scale demands
it) with alerting on any non-zero mismatch count — turning "we found out a
week later" into "we found out within an hour."

### Correction, without downtime

Never edit `wallets.balance` directly, and never delete/edit the erroneous
ledger rows (that would destroy the audit trail and make the bug
un-provable later, e.g. for a support dispute). Instead:

1. Identify affected players and the excess amount per player via the
   ledger (sum of erroneous credits, distinguishable by `reason` if the bug
   is scoped to a specific code path, or by timestamp range matching the
   deploy window of the buggy release).
2. Insert a **compensating ledger entry** per affected player: a negative
   `delta` with `reason = 'correction:duplicate_credit_bug_2026-06-XX'`,
   applied via the same atomic conditional-`UPDATE` pattern used for normal
   debits (so it can't take a wallet negative if the player already spent
   the erroneous funds — that edge case needs a manual/support-reviewed
   queue rather than an automatic negative balance).
3. Run this as a script/migration, in batches, fully logged — itself
   producing new ledger rows, so the correction is as auditable as the
   original bug.

### What invariant would have caught it sooner

The reconciliation query above, run continuously from day one, is the
single invariant that would have caught this within the first
reconciliation cycle after the bad deploy, rather than a week later via
(presumably) a support ticket or manual investigation. The deeper lesson: an
authoritative balance column with no independent ledger to check it against
has no way to self-detect drift — the ledger's entire value is as a
cross-check, not just a historical record.
